# Folder Structure & Conventions

## Naming Rules

- **Files:** camelCase — `workoutLog.ts`, `programCard.tsx`
- **Components:** PascalCase — `WorkoutCard.tsx`, `SetLogger.tsx`
- **Hooks:** camelCase prefixed with `use` — `useAuth.ts`
- **Services:** camelCase suffixed with `.service.ts` — `program.service.ts`
- **Stores:** camelCase suffixed with `.store.ts` — `auth.store.ts`
- **Database:** snake_case at Supabase level (auto-generated types handle the bridge)

---

## Folder Rules

### `app/` — Navigation only

### `src/components/` — UI only

- `ui/` — primitive components: Button, Card, Input, Typography
- `workout/`, `programs/`, etc. — domain-specific components
- `common/` — LoadingScreen, EmptyState, ErrorBoundary
- No data fetching inside components — use hooks

### `src/hooks/` — Data fetching and local state

- Hooks call services, never Supabase directly
- Named `use[noun].ts`

### `src/services/` — All Supabase queries

- Every database query lives here and nowhere else
- One file per entity: `program.service.ts`, `workoutLog.service.ts`, etc.
- If you are writing a Supabase query outside of `services/`, stop and move it here

### `src/stores/` — Global state (Zustand)

- `auth.store.ts` — current user and session
- `workoutSession.store.ts` — active in-progress workout state

### `src/lib/` — Third-party client setup

- `supabase.ts` — single Supabase client instance, imported everywhere
- `sentry.ts` — Sentry init
- `storage.ts` — Supabase Storage helpers

### `src/types/`

- `database.types.ts` — auto-generated by Supabase CLI, never edit manually
- `app.types.ts` — app enums and custom types
- Regenerate database types with: `supabase gen types typescript`

### `src/constants/`

- `theme.ts` — all design tokens (colors, spacing, typography)
- `config.ts` — app-wide constants

### `supabase/`

- `migrations/` — SQL migration files
- `functions/` — Edge Functions if needed
